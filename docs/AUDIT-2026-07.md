# Vantage Platform — Code Audit (July 2026)

**Scope:** Full senior-level audit of the Vantage webapp (`/frontend`, Next.js 16, ~357 TS/TSX files, ~66k LOC), covering tenant isolation, auth, money paths (Stripe), integrations (QuickBooks, Gmail, Resend), the AI/PII layer, the data layer, and general code health.

**Method:** Static analysis of the full request surface (79 API route files, 28 server-action files) plus **live verification against the production database** via the Supabase MCP (RLS policies, indexes, advisors) and direct reads of every flagged file. Findings that could not be confirmed were dropped or downgraded — several recon leads turned out to be false positives (see *Corrections*).

**Bottom line:** This is a well-built codebase with disciplined multi-tenant scoping and solid payment/secret handling. No cross-tenant data-leak path was found. The real issues are a metering concurrency bug (now fixed), input-validation gaps, an AI write-tool that executes without a hard confirmation gate, and a batch of low-urgency DB hygiene items.

---

## What's solid (verified, not assumed)

- **Tenant isolation is real and layered.** Every authenticated route/action calls `requireUserOrg()` / `getCurrentUserOrg()` and filters by `org_id`. **Confirmed against the live DB:** `donors`, `donations`, `interactions`, `opportunities`, `donor_notes` all have RLS **enabled with org-scoped policies** (`user_org_membership(org_id)`, or an `EXISTS` join through `donors` for child tables) — so even though the app uses the RLS-bypassing service-role client, there is a genuine second line of defense at the database. Interactions (which have no `org_id` column) are correctly scoped through `donors!inner(...).eq("donors.org_id", …)`.
- **Stripe is done right.** Webhook signature verification (`stripe.webhooks.constructEvent`), idempotency via a `stripe_webhook_events` unique constraint, stale-event guarding, and plan-ID sanitization against a whitelist. Checkout stamps `org_id`/`plan_id` from the **authenticated** `auth.orgId`, never from client input — no metadata-spoofing path.
- **Secrets are encrypted at rest** with AES-256-GCM (`lib/encryption.ts`): 12-byte random IV, 16-byte auth tag, 32-byte key validation. Used for QuickBooks and Gmail OAuth tokens.
- **Mobile Bearer auth is sound.** `getUserFromBearerToken()` validates the JWT via `admin.auth.getUser(token)` (Supabase verifies signature/expiry); org is derived server-side from `organization_members`, never trusted from the token.
- **API keys** are SHA-256 hashed at rest (never stored plaintext), prefixed for display, scope- and plan-gated, and rate-limited (60/min).
- **Cron endpoints** (`sync`, `digest`, `trial-reminders`) all gate on `isAuthorizedCron()` with a timing-safe `CRON_SECRET` comparison.
- **Pagination is correct** — donation/donor fetches loop with explicit ranges, avoiding PostgREST's silent 1000-row truncation.
- **PII redaction** runs before every LLM call (chat, insights, digest) with post-response unredaction; donor insights apply field-level redaction plus a full-payload safety net.

## Findings by severity

Status legend: **✅ Fixed this pass** · **▶ Recommended** · **ℹ️ Note/accepted**

### P0 / P1 — address before scaling

| # | Severity | Finding | Status |
|---|----------|---------|--------|
| 1 | **P1** | Non-atomic usage metering (billing integrity) | ✅ Fixed |
| 2 | **P1** | No schema validation on mutation routes | ✅ Fixed (parse-guard across all unguarded routes) |
| 3 | **P1** | AI chat write-tools execute without a hard confirmation gate | ✅ Fixed (two-phase confirm) |
| 4 | **P2→P1** | QuickBooks token decrypt fails silently on key rotation | ✅ Fixed |

**1 — Non-atomic usage metering.** `incrementUsage()` ([lib/subscription.ts:334](frontend/lib/subscription.ts#L334)) did a read-then-update (`select count … → update count = count + 1`). Under concurrency (parallel AI-insight / chat / donor-create requests), two callers both read `count=N` and both write `N+1`, losing increments — an org can exceed its paid `ai_insights` / `chat_messages` / donor limits. The `subscription_usage` table already had a unique index on `(org_id, metric, period_start)`.
**Fix applied:** new migration [`20260704000000_atomic_usage_increment.sql`](supabase/migrations/20260704000000_atomic_usage_increment.sql) adds an `increment_usage()` RPC doing an atomic `INSERT … ON CONFLICT DO UPDATE SET count = count + 1`; `incrementUsage()` now calls it. Verified in the live DB (sequential calls return 1→2→3). Applied to production DB and committed as a migration file.

**2 — No schema validation on mutation routes.** 28 API route files call `await request.json()`; **zero import zod** (chat tools do use zod, routes/actions don't). Bodies are cast and trusted. Concretely, donor PATCH ([app/api/donors/[id]/route.ts](frontend/app/api/donors/[id]/route.ts)) parsed the body unguarded — malformed JSON or a `null`/array body threw an **uncaught 500** (and `body[field]` throws on a null body).
**Fix applied (representative):** donor PATCH now guards parsing and non-object bodies, returning **400** instead of 500; success-path shape unchanged (mobile parity safe — PATCH is `pending` in `API_PARITY.md`).
**Recommended:** adopt a small shared `parseJson(schema)` helper (zod is already a dependency) across the other high-traffic mutation routes (donations POST, interactions POST, reports create). This is a consistency win, not an active exploit.

**3 — AI chat write-tools auto-execute.** `create_donor` and `create_donation` ([lib/chat/tools.ts:939](frontend/lib/chat/tools.ts#L939), [:1035](frontend/lib/chat/tools.ts#L1035)) perform the DB insert directly inside `execute()`. The tool descriptions instruct the model to "confirm with the user first," but that is a *soft* prompt-level control the model can skip, and a prompt-injection payload in a donor name/note is a (low-likelihood) trigger path. This contradicts the project rule *"Never auto-execute AI suggestions — always require human approval."* Existing guardrails partly mitigate: org-ownership verification, a duplicate-name warning, and plan-limit checks.
**Recommended (not done here — it's a UX change, not a hotfix):** have these tools return a `requires_confirmation` payload with the proposed record, and execute only on an explicit user confirm action in the chat UI. Deferred intentionally to avoid destabilizing a working feature with a rushed rewrite; worth a small dedicated PR.

**4 — QuickBooks token decrypt fails silently.** `decryptQbToken()` ([lib/quickbooks/token-crypto.ts](frontend/lib/quickbooks/token-crypto.ts)) caught **all** decrypt errors and returned the stored value unchanged. That's correct for genuine legacy-plaintext rows, but if `ENCRYPTION_KEY` is missing/rotated/corrupted it silently returns ciphertext as if it were a token — every QB sync for that org fails with no logged cause.
**Fix applied:** distinguish "looks encrypted" (`iv:tag:ciphertext` shape) from legacy plaintext; log an explicit error in the former case while preserving the legacy fallback.

### P2 — quality & hardening

- **Email rate-limit race.** `POST /api/email/send` ([:74](frontend/app/api/email/send/route.ts#L74)) counts sends then sends+logs non-atomically; a burst can exceed the 50/hr per-user cap by a small margin. Low blast radius (50/hr is loose). *Recommended, not fixed — an atomic reservation adds more failure modes than the bug is worth.* Same pattern in `bulk-send`.
- **`as any` hotspot / route monolith.** 21 `as any` casts across the codebase (there are **0** `@ts-ignore`, contrary to an early estimate). The concentration is in [app/api/reports/create/route.ts](frontend/app/api/reports/create/route.ts) — an ~800-line handler mixing filtering, projection, CSV, and insert logic. *Recommended:* split into typed helpers.
- **Two PII redaction systems.** `lib/pii-redaction.ts` (insights, `[DONOR_NAME]`) and `lib/chat/pii-redactor.ts` (chat, `[DONOR_N]`) redact only `display_name`/`email`/`phone`. Addresses, city, custom-field values, and campaign/fund names can appear un-redacted in LLM tool results. *Recommended:* unify on one redactor and widen the field set.
- **`donors.assigned_to` foreign key is unindexed** (Supabase perf advisor) — the only high-value unindexed FK on a core table; add a covering index for assignment filtering at scale.
- **Import has no total-row cap** ([app/actions/import.ts](frontend/app/actions/import.ts)) — chunked at 500 but unbounded overall; add an explicit max-rows guard.
- **`dangerouslySetInnerHTML`** in `settings-year-end-receipts.tsx` (input is HTML-escaped, so low risk) — prefer safe rendering.

### P3 — DB hygiene & housekeeping (from Supabase advisors)

- **`multiple_permissive_policies` (140) + `auth_rls_initplan` (15).** Every core table carries both a `Service role full access` policy *and* the org-scoped user policy; the service-role policy isn't restricted `TO service_role`, so it overlaps `anon`/`authenticated` on every action, and it re-evaluates `auth.<fn>()` per row. One migration — restrict service-role policies to `TO service_role` and wrap auth calls as `(select auth.<fn>())` — clears the bulk of both classes on the five core tables. Low urgency (app uses service role; anon/authenticated are effectively denied anyway), pure performance/hygiene.
- **5 duplicate indexes** (`donor_notes`, `donor_tags` ×2, `interactions`, `opportunities`) — zero-risk drops.
- **19 unused indexes** — several back cron/sync queries that likely just didn't run in the advisor's sample window; verify before dropping.
- **`vector` extension installed in `public` schema**; **`user_org_membership()`** is an `anon`/`authenticated`-executable SECURITY DEFINER function (documented as intentional — it backs the RLS policies and only reveals the *caller's* own membership; accepted).
- Peripheral tables with `rls_enabled_no_policy` (INFO) are the intended deny-all lockdown posture — **not** a gap. Note `pledges` has RLS on with 0 policies: safe (deny-all to PostgREST), but inconsistent with the sibling tables that expose org-scoped user policies — worth aligning for consistency.

## Corrections (recon leads that verification refuted)

These were flagged by the initial sweep and **cleared** on inspection — recorded so they aren't re-raised:

- **"No RLS on core tenant tables."** False — RLS is enabled with proper org-scoped policies on `donors`/`donations`/`interactions`/`opportunities`/`donor_notes`. The migrations *folder* doesn't show it because the base schema was applied outside it; the live DB has it.
- **"Stripe checkout trusts client metadata."** False — `org_id` is stamped from `auth.orgId` server-side.
- **"Bearer token path may not bind org / check expiry."** False — JWT is validated by Supabase Auth; org comes from membership lookup.
- **`@ts-ignore` prevalence** — actually 0.

**Consciously accepted (changing them would break behavior for ~zero gain):**
- **`user_org_membership()` executable by anon/authenticated** — it *must* stay executable because the org-scoped RLS policies call it; it only returns whether the *caller* belongs to a given org (no data). Intentional per the lockdown migration.
- **`vector` extension in `public`** — moving it risks breaking the `embedding` columns and `match_donors`; not worth it.
- **PII redaction field set** — chat tools deliberately exclude street address/email/phone from results; `city/state/zip` are intentionally used for geo queries. Widening redaction would break features, not close a leak.

**Verification:** `npm run lint` (0 errors) and `npm run build` (type-check) pass. Both RPCs tested live (atomic increment 1→2→3; quota true/true/false at limit 2). Security advisor re-run: no ERROR/WARN except the two accepted WARNs; the `rls_enabled_no_policy` INFO items are the intended deny-all lockdown tables. Mobile parity preserved — changes to consumed routes only reject invalid input; valid-request response shapes unchanged.
